Урок 5 · Next.js — рендеринг и кеш

Кеш и revalidate: когда страница свежая, а когда нет

Ты только что в merge добавил export const revalidate = 0; в categories/[slug]/page.tsx. Эта строчка стоит на каждой витринной странице проекта. Разберём, что она реально делает — и почему без неё страница может «залипнуть» со старой ценой товара.

Цель урока. Понять, что Next по умолчанию кеширует результат рендера страницы, и научиться управлять свежестью тремя ручками: revalidate = 0 (всегда динамика), revalidate = N (раз в N секунд) и on-demand revalidatePath() (точечный сброс по событию). Всё на реальном коде payload-start.
Мост от ExtJS. В ExtJS приложение живёт в браузере: открыл грид — он store.load()'нул свежие данные с сервера прямо сейчас. «Кеша рендера» там нет — каждый показ это живой запрос. Next переворачивает это: он старается отрендерить страницу один раз и потом раздавать готовый результат как статику. Быстро для пользователя — но именно поэтому появляется вопрос «а когда обновить?». revalidate — ответ на этот вопрос.

1. Главная идея: static by default

Когда ты делаешь npm run build, Next смотрит на каждую страницу и пытается отрендерить её заранее, на этапе сборки, и сохранить результат (HTML + RSC-payload). Дальше пользователям отдаётся этот готовый снимок — без захода в БД на каждый запрос.

Это и есть два режима рендеринга:

РежимКогда рендеритсяДанныеСкорость
Static (по умолчанию)Один раз — на билде (или при revalidate)«Замороженные» на момент рендераОчень быстро (отдаётся готовое)
DynamicНа каждый запрос пользователя (SSR)Всегда свежие из БДМедленнее (рендер + запрос каждый раз)

В билде, который мы только что прогнали, ты это видел в легенде:

○  (Static)   prerendered as static content
ƒ  (Dynamic)  server-rendered on demand

Все наши /products, /categories, /contacts были помечены ƒ — динамические. Не случайно: мы их такими сделали. Разберём чем.

2. Что переключает страницу в Dynamic

Next сам решает режим. Страница становится динамической, если в ней есть хоть один «динамический» сигнал:

ТриггерПочему
cookies() / headers()Запросо-зависимы — на билде их нет
searchParams в pageЗависят от конкретного URL запроса
export const dynamic = 'force-dynamic'Явный приказ «всегда SSR»
export const revalidate = 0«Кеш живёт 0 секунд» = по сути всегда заново
fetch с { cache: 'no-store' }Запрос не кешируется → и страница тоже
Тонкость про Payload Local API. Мы данные тянем не через fetch(), а через payload.find() (прямой вызов в Server Component). Next не умеет автоматически отслеживать такой источник для кеша — он про него ничего не знает. Поэтому без явного revalidate = 0 страница со списком товаров могла бы закешироваться на билде и показывать старый список, пока ты не передеплоишь. Вот зачем эта строчка стоит везде.

3. Ручка export const revalidate

Один экспорт на уровне модуля страницы/layout'а. Три осмысленных значения:

ЗначениеСмыслКогда брать
false (дефолт)Static «навсегда» — до следующего билда или on-demand сбросаКонтент не меняется (лендинг, статья)
0Всегда Dynamic — рендер на каждый запрос, всегда свежоДанные часто меняются и важна свежесть «сейчас»
3600 (любое N)ISR: отдаём кеш, но не старше N секунд; устарел — Next тихо перерендерит в фонеМожно потерпеть задержку в N сек ради скорости

Наш выбор по проекту — 0 на всех витринных страницах:

// src/app/(frontend)/products/page.tsx, categories/page.tsx,
// categories/[slug]/page.tsx, contacts/page.tsx — везде одинаково:
export const revalidate = 0;

Перевод: «цена, наличие, баннер магазина могут поменяться в админке в любой момент — рендери на каждый заход, не показывай старьё». Платим скоростью, покупаем свежесть.

4. On-demand revalidation: push вместо poll

revalidate = N — это poll: «проверяй устаревание по таймеру». Но есть и push: «данные изменились — сбрось кеш именно сейчас, точечно». Это revalidatePath() / revalidateTag() из next/cache.

В проекте уже собрана полная цепочка такого сброса. Триггер — сохранение товара в админке Payload:

// src/hooks/products/afterChange.ts — хук Payload после сохранения товара
export const afterChange: CollectionAfterChangeHook = async ({ doc, req, operation }) => {
  if (doc._status !== 'published') return;        // черновики не трогаем
  if (req.context?.isSeeding) return;            // при сидинге веб-сервера нет — пропускаем

  await fetch(`${process.env.NEXT_PUBLIC_API_URL}/api/revalidate`, {
    method: 'POST',
    headers: { 'x-revalidate-secret': process.env.REVALIDATE_SECRET ?? '' },
  });
};

Хук стучится на свой же API-роут. А роут — проверяет секрет и дёргает сброс:

// src/app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache';

export async function POST() {
  const headers = await getHeaders();
  if (headers.get('x-revalidate-secret') !== process.env.REVALIDATE_SECRET) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
  revalidatePath('/products');              // сбросить кеш страницы /products
  return NextResponse.json({ revalidated: true, now: Date.now() });
}
Сохранил товар в админке └─► products/afterChange (Payload hook) └─► POST /api/revalidate + секрет в заголовке └─► проверка секрета ─► revalidatePath('/products') └─► кеш страницы помечен устаревшим ─► след. заход = свежий рендер

Зачем секрет? /api/revalidate — публичный URL. Без проверки любой смог бы спамить сбросом кеша (мелкий DoS). Секрет в заголовке = «свой/чужой».

Острый вопрос — подумай прежде чем читать спойлер. Если все витринные страницы стоят на revalidate = 0 (то есть и так рендерятся заново на каждый запрос, кеша нет)… то зачем тогда нужна вся эта on-demand цепочка с revalidatePath('/products')? Что она реально сбрасывает?
Спойлер: разбор несостыковки

При revalidate = 0 сбрасывать почти нечего — страница и так динамическая. То есть прямо сейчас эти два механизма дублируют цель (свежесть), и on-demand-хук в значительной мере холостой. Это нормальная находка для mid-level код-ревью. Возможные трактовки:

  • Исторический слой: хук revalidate писался раньше, когда страницы были static/ISR, а revalidate = 0 добавили позже «чтобы наверняка» — и забыли убрать хук.
  • Задел на будущее: если когда-нибудь увести /products в ISR (revalidate = 3600) ради скорости — on-demand сброс сразу станет нужен, чтобы новый товар появлялся мгновенно, не дожидаясь часа.
  • Правильный дизайн «как надо»: страницы делают static/ISR (быстро), а свежесть обеспечивает on-demand сброс по событию из CMS. Это и есть рекомендуемый Next-паттерн для headless-CMS. Текущий проект до него «не дотянул» — выбрал грубое revalidate = 0.

Вывод для собеседования: уметь сказать «revalidate=0 просто и надёжно, но жертвует скоростью; зрелый вариант — ISR + on-demand revalidate из хука CMS».

5. revalidatePath vs revalidateTag

APIСбрасываетКогда
revalidatePath('/products')Конкретный маршрут по URLЗнаешь, какая страница протухла
revalidateTag('products')Все fetch'и, помеченные тегом products, где бы они ни былиДанные размазаны по многим страницам

Теги вешаются на запрос: fetch(url, { next: { tags: ['products'] } }). В нашем проекте используется path-вариант — он проще и нам хватает (одна «горячая» страница списка). Tag-вариант пригодится, когда товар светится на десятке разных страниц.

3. Зачем в products/afterChange.ts стоит if (req.context?.isSeeding) return; перед вызовом /api/revalidate?

4. Хочешь: страница быстрая (static), но новый товар появляется на ней сразу после сохранения в админке. Какой подход?

Практика 3 (пишешь сам)

Закрепляем на реальном коде проекта. Задание — небольшое, но «сквозное»:

  1. Выбери страницу детали товара src/app/(frontend)/products/[id]/page.tsx. Сейчас на ней нет revalidate — проверь это.
  2. Переведи её на ISR: добавь export const revalidate = 60;. Запусти npm run build и найди страницу /products/[id] в выводе — как она помечена (○/ƒ/●)? Объясни словами, что изменилось.
  3. Теперь дополни on-demand цепочку: в api/revalidate/route.ts рядом с revalidatePath('/products') подумай, как сбросить и конкретную карточку товара. Набросай (можно псевдокодом в комментарии), что бы ты передал, чтобы сбросить /products/42. Подсказка: revalidatePath принимает путь.
  4. Откати revalidate = 60 обратно если хочешь сохранить текущее поведение — главное, чтобы ты понял разницу на собственном билде.
