Урок 3 · Next.js App Router

Папки становятся URL

Ты только что создал categories/page.tsx — и страница появилась на /categories. Никакой регистрации маршрута. Разберём систему, которая это делает: файловая структура — это и есть роутер.

Что унесёшь. Сможешь по любому файлу проекта сказать, на каком URL он живёт и что именно рендерит — и наоборот, по URL найти нужный файл. Это база навигации по любому Next-проекту.
Мост от ExtJS. В ExtJS маршруты ты регистрировал руками — hash-роуты, контроллеры, Ext.util.History. В App Router маршрутов как сущности нет: есть дерево папок. Создал папку с page.tsx — URL появился. Удалил — исчез. Роутер выводится из ФС.

1. Правило №1: папка = сегмент URL, page.tsx = страница

Каждая папка внутри src/app/ добавляет сегмент к адресу. URL рисует только файл с зарезервированным именем page.tsx — просто папка без него маршрутом не становится.

Файл в проектеURL
(frontend)/page.tsx/
(frontend)/categories/page.tsx/categories ← твой
(frontend)/cart/page.tsx/cart
(frontend)/orders/page.tsx/orders

2. Зарезервированные имена файлов

App Router узнаёт лишь несколько имён. Всё остальное (AddToCartButton.tsx, SearchInput.tsx) — обычные модули, на роутинг не влияют, лежат рядом спокойно.

ФайлРольПример в проекте
page.tsxUI маршрута (то, что видно на этом URL)products/page.tsx
layout.tsxОбёртка вокруг страниц; переживает навигацию, не перемонтируется(frontend)/layout.tsx
route.tsHTTP-эндпоинт (GET/POST…), а не UI. Не сочетается с page.tsx в одной папкеapi/checkout/route.ts
loading.tsxФолбэк, пока грузится page (через Suspense)позже
error.tsxUI при выброшенной ошибке в сегментепозже
Ключевое отличие layout от page: при переходе с /products на /products/5 общий layout.tsx не размонтируется — его состояние, скролл, открытые панели сохраняются. Меняется только вложенный page. Это и есть причина, почему layout кладут отдельно, а не копируют шапку в каждую страницу.

3. Route groups: (frontend) и (payload)

Папка в круглых скобках — организационная. Она группирует маршруты и даёт им отдельный layout.tsx, но в URL не попадает.

// (frontend)/categories/page.tsx
URL = /categories      ← «(frontend)» вырезано

// (payload)/admin/[[...segments]]/page.tsx
URL = /admin/...       ← «(payload)» вырезано

Зачем так в этом проекте: витрина и админка — два разных мира с разными обёртками. (frontend)/layout.tsx даёт шапку магазина, CartProvider, тему. (payload)/layout.tsx — совсем другой каркас Payload-админки. Скобки позволяют держать два независимых корневых layout'а бок о бок, не плодя сегмент в адресе.

4. Динамические сегменты: [id]

Папка в квадратных скобках ловит любое значение и отдаёт его странице как параметр.

// (frontend)/products/[id]/page.tsx
/products/5    → params.id === "5"
/products/abc  → params.id === "abc"

export default async function ProductDetail({ params }) {
  const { id } = await params;   // в Next 15 params — Promise
  // ...payload.findByID({ collection:'products', id })
}

В проекте так устроены products/[id] и orders/[id] — одна папка-шаблон обслуживает тысячи товаров/заказов.

Соседи [id] — catch-all (для справки, не учим сейчас)
СинтаксисЛовитВ проекте
[id]ровно один сегментproducts/[id]
[...slug]один и более сегментовapi/[...slug]/route.ts
[[...segments]]ноль и более (опционально)admin/[[...segments]]

Payload подключает всю свою админку и REST API одной catch-all папкой — поэтому /admin, /admin/collections/products и т.д. все ведут в один page.tsx.

5. Вложенные layout'ы складываются

Layout'ы по пути оборачиваются друг в друга сверху вниз. Для /products/5:

(frontend)/layout.tsx ← шапка, CartProvider (вся витрина) └ products/layout.tsx ← sidebar-каркас (только /products*) └ [id]/page.tsx ← карточка товара

Итог рендера = frontend layout → внутри products layout → внутри page. Поэтому твой categories/page.tsx получил шапку магазина даром: он под (frontend)/layout.tsx. Но sidebar от products/layout.tsx он не получил — тот живёт глубже, только для ветки /products.

Заметил products/@sidebar/? Папка с @ — это parallel route (именованный слот, рендерится одновременно с основным page). Продвинутая штука — разберём отдельным уроком. Сейчас просто знай: @ ≠ обычный сегмент, в URL он тоже не попадает.

2. Почему у проекта два файла layout.tsx верхнего уровня — в (frontend) и в (payload)?

3. Пользователь переходит с /products на /products/7. Что произойдёт с products/layout.tsx (sidebar-каркас)?

4. В папке api/checkout/ лежит route.ts. Можно ли в ту же папку положить page.tsx?
