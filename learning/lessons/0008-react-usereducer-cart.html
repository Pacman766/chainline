useReducer: состояние как машина переходов

Корзина — это не одно значение, а набор переходов: добавить, убрать, сменить количество, очистить, восстановить из storage. Когда переходов много и они затрагивают один и тот же массив — useState расползается по компоненту. useReducer собирает всю логику изменения в одну чистую функцию. У тебя в проекте это уже сделано — cartReducer.ts + CartContext.tsx. Разбираем построчно.

Цель. Понять три вещи: (1) reducer = чистая функция (state, action) => newState — почему чистая и почему иммутабельная; (2) action как discriminated union — как один тип события даёт TS-исчерпывающий switch без ошибок; (3) как dispatch заворачивается в action creators и раздаётся через Context, и почему totalPrice не лежит в state, а выводится.

1. Обзор: useState vs useReducer

Мост от ExtJS. В ExtJS Ext.data.Store — это состояние + методы-переходы: store.add(), store.remove(), store.removeAll(), и события datachanged. useReducer — тот же паттерн, разнятый на части: state (массив items), action (описание «что произошло» — как событие стора), reducer (как состояние отвечает на событие — как внутренняя логика add/remove). Разница: reducer — чистый (ничего не мутирует, не лезет в сеть), а возвращает новый массив. Стор ExtJS мутировал себя на месте; здесь — иммутабельность.

Когда что:

ПризнакuseStateuseReducer
Сколько переходов1–2 простыхмного (add/remove/update/clear/init)
Зависят ли переходы от текущего stateредкода (add суммирует с тем, что уже в корзине)
Где живёт логикаразмазана по хендлерамв одном месте — reducer
Тестируемостьнужен рендерreducer — чистая функция, тест без React

Правило: как только «обновление состояния» становится логикой (а не просто «положить значение») и переходов больше двух — это reducer. Корзина — хрестоматийный случай.

2. Разбор reducer'а со сносками — cartReducer.ts

Сердце фичи. Цифры [n] — сноски под кодом.

import { CartAction, CartItem } from '@/types/cart';

export function cartReducer(state: CartItem[], action: CartAction): CartItem[] {   // [1]
  switch (action.type) {                                                      // [2]
    case 'ADD': {
      const exists = state.find((p) => p.productId === action.product.productId);  // [3]
      return exists
        ? state.map((p) =>                                                    // [4]
            p.productId === action.product.productId
              ? { ...p, quantity: p.quantity + action.product.quantity }
              : p,
          )
        : [...state, action.product];                                        // [5]
    }
    case 'REMOVE': {
      return state.filter((p) => p.productId !== action.productId);            // [6]
    }
    case 'CLEAR': {
      return [];
    }
    case 'INIT': {
      return action.items;                                                   // [7]
    }
    case 'UPDATE_QUANTITY': {
      if (action.quantity <= 0) {                                          // [8]
        return state.filter((p) => p.productId !== action.productId);
      }
      return state.map((p) =>
        p.productId === action.productId ? { ...p, quantity: action.quantity } : p,
      );
    }
  }                                                                          // [9] — нет default. почему ок?
}
  1. Сигнатура reducer'а фиксирована React'ом: (state, action) => newState. state: CartItem[] — текущая корзина, action: CartAction — описание события. Возвращаемый тип тот же, что у state (CartItem[]) — reducer всегда отдаёт состояние того же вида. Никаких аргументов «извне», никакого await, никакого localStorage внутри: чистая функция.
  2. switch (action.type) — это discriminated union в действии. CartAction (в types/cart.ts) — это |-объединение, где общее поле type — дискриминант. Как только TS видит case 'ADD', он сужает тип action до варианта с product; в case 'REMOVE' — до варианта с productId. Поэтому в ветке REMOVE нет action.product — TS бы не дал. Один тип события заменяет россыпь несвязанных аргументов.
  3. Зависимость от текущего state. Добавляя товар, мы сначала смотрим — он уже в корзине? Эту логику нельзя выразить через useState(setItems(x)) чисто — нужен доступ к актуальному state. Reducer его получает первым аргументом всегда «свежим». (Это та же причина, по которой в setState с зависимостью от прошлого используют функцию-апдейтер.)
  4. Иммутабельное обновление #1 — map. Товар уже есть → возвращаем новый массив, где у совпавшего элемента quantity увеличен. { ...p, quantity: ... } — новый объект, остальные p переиспользуются как есть. Мы НИКОГДА не пишем p.quantity += ... — мутация старого state сломает сравнение по ссылке, и React может не перерисовать (или перерисовать лишнее).
  5. Иммутабельное обновление #2 — спред. Товара нет → [...state, action.product]: новый массив = старые элементы + новый. Снова новая ссылка на массив — сигнал React'у «состояние изменилось».
  6. filter = удаление иммутабельно. Не splice (мутирует), а filter, который возвращает новый массив без удалённого. Паттерн: любое изменение массива в reducer — через map/filter/спред, не через мутирующие методы.
  7. INIT — гидрация из localStorage. Эта ветка просто заменяет весь state на пришедший массив. Кто и когда её шлёт — увидишь в провайдере (сноска про первый useEffect). Важно: reducer не знает про localStorage — он только принимает готовый массив. Грязь (парсинг, try/catch) — снаружи, в эффекте. Reducer чист.
  8. Бизнес-правило внутри перехода. «Снизил количество до 0 или меньше → это удаление». Логика живёт в reducer, а не в компоненте: любой, кто шлёт UPDATE_QUANTITY с quantity:0, получит удаление — единый источник правды. Компоненту не надо помнить про этот кейс.
  9. Нет default — и это корректно сейчас. Все 5 вариантов union'а разобраны, TS считает switch исчерпывающим. Но это хрупко: добавишь 6-й тип в CartAction и забудешь case — функция тихо вернёт undefined для него (тип результата станет CartItem[] | undefined). В практике добавим явную защиту-страж. Запомни запах: switch по union без exhaustiveness-guard.

Тип события — types/cart.ts (источник дискриминанта)

export type CartAction =
  | { type: 'ADD'; product: CartItem }
  | { type: 'REMOVE'; productId: string }
  | { type: 'CLEAR' }
  | { type: 'INIT'; items: CartItem[] }
  | { type: 'UPDATE_QUANTITY'; productId: string; quantity: number };

Каждый вариант несёт ровно те поля, что нужны переходу: ADD — весь товар, REMOVE — только id, CLEAR — ничего. Это и есть «типизированное событие». Сравни с ExtJS, где fireEvent('add', record) — аргументы нетипизированы и проверяются только в рантайме.

Как reducer подключён и обёрнут — CartContext.tsx

const [items, dispatch] = useReducer(cartReducer, []);                // [A] state + диспетчер, старт []

const totalItems = useMemo(() => items.reduce((a,p)=>a+p.quantity,0), [items]);   // [B] выводим, не храним
const totalPrice = useMemo(() => items.reduce((a,p)=>a+p.price*p.quantity,0), [items]);

useEffect(() => {                                                      // [C] гидрация ОДИН раз
  try {
    const cart = localStorage.getItem('cart');
    if (cart) dispatch({ type: 'INIT', items: JSON.parse(cart) });
  } catch { localStorage.removeItem('cart'); }
  setInitialized(true);
}, []);

useEffect(() => {                                                      // [D] персист при изменениях
  if (initialized) localStorage.setItem('cart', JSON.stringify(items));
}, [items, initialized]);

const addItem = (product) => dispatch({ type: 'ADD', product });    // [E] action creator
  • [A] useReducer(reducer, initialState) возвращает [state, dispatch]. dispatch(action) — единственный способ изменить state: он зовёт cartReducer(currentItems, action) и кладёт результат как новый state. Никто не трогает items напрямую.
  • [B] Выводимое состояние не хранят. totalItems/totalPrice — это производные от items. Класть их в reducer-state — значит держать два источника правды и ловить рассинхрон. Считаем на лету через useMemo (мемо — чтобы не пересчитывать на каждый рендер, только при смене items). Связь с Уроком 7: totalPrice здесь — для отображения; «настоящую» сумму заказа всё равно пересчитывает сервер (твой orders/beforeChange), клиенту не верим.
  • [C] и [D] — почему два эффекта и флаг initialized. Эффект [C] поднимает корзину из localStorage один раз (пустой deps []). Эффект [D] сохраняет корзину при каждом изменении items. Флаг initialized критичен: без него [D] выстрелит на первом рендере (когда state ещё [], до гидрации) и затрёт сохранённую корзину пустотой. Порядок: смонтировались с [] → [C] прочитал storage → INIT → initialized=true → только теперь [D] начинает писать. Это типовая SSR/гидрация-ловушка: localStorage нельзя читать при рендере (на сервере его нет) — только в эффекте.
  • [E] Action creators прячут dispatch. Наружу через Context уходят не «сырой» dispatch, а удобные addItem/removeItem/.... Компонент пишет addItem(product), не зная формы экшена. Это инкапсуляция: форма action'а — деталь, её можно поменять, не трогая UI.
Три ловушки reducer'а:
• Мутация state. state.push(x); return state — классика бага: ссылка та же, React может пропустить ререндер. Всегда новый объект/массив (map/filter/спред).
• Побочка в reducer. fetch, localStorage, Date.now() внутри reducer'а — нельзя. Reducer обязан быть чистым и предсказуемым (React в Strict Mode зовёт его дважды — побочка задвоится). Грязь — в эффект или в action creator.
• Выводимое в state. Не клади totalPrice в reducer-state. Если значение можно посчитать из другого — считай, не храни.

4. Практика 6 (пишешь сам)

Задача с реальным смыслом: на странице корзины кнопки «+ / −» сейчас вынуждены сами считать quantity ± 1 и звать updateQuantity(id, q+1) — то есть знать текущее количество. Перенесём логику шага в reducer: компонент будет слать намерение («увеличь на 1»), а не вычисленное значение. Плюс закроем хрупкость из сноски [9].

  1. Расширь тип. В types/cart.ts добавь в CartAction два варианта: { type: 'INCREMENT'; productId: string } и { type: 'DECREMENT'; productId: string }.
  2. Добавь exhaustiveness-guard. В конец switch в cartReducer.ts добавь default, который заставит TS падать на необработанном варианте: 
    default: {
  const _exhaustive: never = action;   // TS-ошибка, если забыл case
  return state;
}
    Сохрани файл с ещё НЕ написанными case'ами для INCREMENT/DECREMENT — посмотри на ошибку TS в _exhaustive (это и есть страж в действии). Объясни словами, почему тип never ловит пропущенный case.
  3. Реализуй переходы. INCREMENT: quantity + 1 у совпавшего товара (иммутабельно, через map). DECREMENT: quantity − 1, но если стало <= 0 — удали товар (переиспользуй правило из UPDATE_QUANTITY, не дублируй мысль бездумно — подумай, можно ли свести). После этого ошибка в _exhaustive должна исчезнуть.
  4. Прокинь action creators. В CartContext.tsx добавь increment(productId) / decrement(productId) рядом с остальными, добавь их в тип CartContextType и в value провайдера.
  5. Используй в UI. На странице корзины кнопки «+/−» переведи на increment(id) / decrement(id). Проверь: «−» на товаре с количеством 1 убирает его из корзины; перезагрузка страницы сохраняет состояние (персист работает).
  6. На подумать (без кода): почему правильно, что «дошёл до 0 → удалить» живёт в reducer, а не в обработчике клика кнопки? Свяжи с идеей «единый источник правды для перехода».
