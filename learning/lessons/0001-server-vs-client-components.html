Server Components vs Client Components

Главный ментальный сдвиг при переходе с ExtJS. Поймёшь его — и 80% этого проекта станут очевидными.

Мост от ExtJS. В ExtJS всё живёт в браузере: ты создаёшь Ext.create('Ext.grid.Panel', …), стор тянет данные по AJAX, виджет рендерит DOM на клиенте. В Next App Router по умолчанию всё наоборот: компонент выполняется на сервере, а в браузер уходит уже готовый HTML. Клиентский JS — это исключение, которое ты включаешь осознанно. Перевернуть эту интуицию — и есть задача урока.

1. Две среды выполнения

В App Router у компонента есть «гражданство» — где он выполняется:

🖥️ Server Component по умолчанию

  • Выполняется на сервере, при запросе страницы
  • Может быть async и ждать данные
  • Прямой доступ к БД, файлам, секретам
  • Отправляет в браузер HTML, не свой JS
  • ❌ Нет useState, onClick, useEffect

🌐 Client Component 'use client'

  • Выполняется в браузере (как привычный React)
  • Состояние, эффекты, обработчики событий
  • ❌ Не async, нет прямого доступа к БД
  • Его код уходит в JS-бандл клиента
  • Включается директивой 'use client' вверху файла

Ключевое правило выбора: нужна интерактивность или браузерный API → Client. Иначе → Server. Не «всё подряд client на всякий случай» — это раздувает бандл и убивает преимущество.

2. Реальный Server Component из проекта

Открой src/app/(frontend)/products/page.tsx. Вот его суть:

// НЕТ 'use client' → это Server Component (по умолчанию)
import config from '@payload-config';
import { getPayload } from 'payload';

export default async function ProductPage() {
  // ⬇ прямой запрос к БД ПРЯМО в компоненте. На клиенте так нельзя.
  const payload = await getPayload({ config });
  const result = await payload.find({ collection: 'products' });

  return (
    <div className="product-grid">
      {result.docs.map((p) => <ProductCard product={p} />)}
    </div>
  );
}

Три вещи, невозможные в ExtJS-виджете и в обычном React-компоненте, но нормальные здесь:

  • Функция компонента — async.
  • await payload.find(...) — поход в базу внутри компонента, без отдельного API-эндпоинта и без AJAX.
  • Секрет/конфиг (config) импортируется прямо сюда — он никогда не утечёт в браузер, потому что этот код туда и не попадёт.
Почему это безопасно. Код Server Component не отправляется клиенту — браузер получает только отрендеренный HTML карточек. Поэтому payload.find, ключи и SQL остаются на сервере. Это и есть причина, по которой RSC вообще придумали.

3. Где включается клиент

Карточка товара — ссылка, её рисует сервер. Но кнопка «В корзину» должна реагировать на клик и менять состояние корзины. Клик — это интерактивность → нужен Client Component. Смотри AddToCartButton.tsx и contexts/CartContext.tsx:

'use client';  // ⬅ граница: этот файл и его импорты — клиентские

import { useReducer, useContext } from 'react';

export function CartProvider({ children }) {
  const [items, dispatch] = useReducer(cartReducer, []);
  // useState/useReducer/onClick живут ТОЛЬКО в client-компонентах
  ...
}

Заметь паттерн композиции: серверная страница рендерит список и внутрь карточек вставляет маленькую клиентскую кнопку. Сервер делает тяжёлую работу с данными, клиент — только точечную интерактивность. Это и есть идиоматичный App Router.

ExtJS-аналогия для интуиции. Представь, что разметку грида отрендерил сервер и прислал готовый HTML, а ты «оживляешь» только отдельные кнопки маленькими клиентскими виджетами. В ExtJS так не делают (там всё клиент), но именно эта гибридная модель — суть Next.

4. Как распознать тип за 2 секунды

  • Есть 'use client' в первой строке файла? → Client.
  • Нет? Функция async и/или ходит в БД/await данных? → Server.
  • Есть useState/onClick/useEffect? → обязан быть Client (иначе ошибка сборки).
