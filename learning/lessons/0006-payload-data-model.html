Урок 6 · Payload — модель данных

Коллекция как единый источник правды

Ты уже читал данные из коллекций (payload.find, findByID) и даже руками собрал глобал SiteSettings с group/array. Теперь зайдём с другой стороны: что такое сам config коллекции и почему один объект на ~30 строк порождает таблицу в БД, экран в админке, REST+GraphQL API и TypeScript-типы — одновременно.

Цель урока. Научиться читать и писать CollectionConfig: slug → fields → access → hooks → admin. Понять типы полей на реальном коде Products.ts/Orders.ts, и главное — увидеть, что access в Payload может вернуть не только true/false, но и фильтр-запрос (построчная безопасность из коробки).
Мост от ExtJS. В ExtJS у тебя Ext.define('App.model.Product', {fields:[...]}) — модель описывает только форму данных на клиенте. Сервер, таблицу в БД, валидацию и права ты пишешь отдельно, руками, на другом стеке. Payload-коллекция — это та же «модель», но она сама себе бэкенд: из одного описания полей Payload поднимает миграцию схемы, CRUD-эндпоинты, проверки доступа и форму редактирования. Один файл вместо четырёх слоёв.

1. Из одного config — четыре артефакта

Возьми Categories.ts — самую маленькую коллекцию проекта, всего два поля:

export const Categories: CollectionConfig = {
  slug: 'categories',
  fields: [
    { name: 'name', type: 'text', required: true },
    { name: 'slug', type: 'text', required: true, unique: true },
  ],
};

Этот объект Payload разворачивает сразу в четыре вещи:

АртефактЧто появляется
Схема БДтаблица categories с колонками name, slug (+ id, createdAt, updatedAt автоматом). unique → UNIQUE-индекс.
Admin UIраздел «Categories» с формой из двух текстовых инпутов и валидацией «обязательно».
APIREST (/api/categories) + GraphQL + Local API (payload.find({collection:'categories'}), которым ты уже пользовался).
Типыинтерфейс Category в src/payload-types.ts после npm run generate:types.

Поменял поле в config → поменялись все четыре. Это и есть «единый источник правды»: нет рассинхрона между БД, формой и типами, потому что они выводятся из одного места.

2. Анатомия коллекции

Полный набор верхнеуровневых ключей CollectionConfig (все, кроме slug/fields, опциональны). Где что в проекте:

КлючЗачемПример в коде
slugидентификатор: имя таблицы + путь APIвсе коллекции
fieldsполя = колонки + инпутывсе
accessправа на read/create/update/deleteProducts.ts:23
hooksкод на события жизненного циклаProducts.ts:29 (beforeChange/afterChange/beforeDelete)
adminкак коллекция выглядит/ведёт себя в админкеProducts.ts:15 (useAsTitle, preview)

Деталь про auth: true. Одна строка в Customers.ts превращает обычную коллекцию в сущность с аутентификацией: Payload добавляет поля email/password (хеш), эндпоинты login/logout/refresh и механику JWT. Поэтому у тебя в проекте две auth-коллекции (users для админки и customers для магазина) — каждая со своим логином, как описано в CLAUDE.md.

3. Типы полей — экскурсия по Products.ts

Карточка товара использует почти весь словарь field types. Главное, что нужно усвоить: тип поля — это не только UI-виджет, это ещё и тип колонки в БД, и форма в TS-типе, и набор доступных валидаторов.

typeЧто этоВ Products.ts
textстрокаname (required)
numberчисло + min/maxprice (min:0, max:1e6)
checkboxbooleaninStock
richTextструктурный документ (не строка!) — рендерится конвертеромdescription
relationshipссылка на другую коллекцию (внешний ключ)category → categories
relationship + hasManyмассив ссылок (many-to-many)images → media, hasMany

Тонкость group vs collapsible. Оба «складывают» поля, но по-разному. group с name:'dimensions' создаёт вложенность в данных → в API ты читаешь product.dimensions.weight (ты это и делаешь на PDP). collapsible — только UI-гармошка в админке, в данных плоско: dimensions и images лежат на верхнем уровне товара, хотя в форме спрятаны под одну шапку. Структура данных определяется group/array, а не визуальными обёртками.

4. Поля тоже программируемые: field-level hooks

Хук можно повесить не только на коллекцию, но и на отдельное поле. Смотри price (Products.ts:50):

{
  name: 'price', type: 'number', required: true, min: 0, max: 1000000,
  hooks: {
    beforeChange: [({ value }) => value === null ? value : Math.round(value * 100) / 100],
    afterRead:    [({ value }) => value ?? 0],
  },
}
  • beforeChange на поле — нормализует перед записью: округляет цену до 2 знаков. Кто бы ни писал (админка, seed, API) — в БД ляжет округлённое.
  • afterRead — чинит данные на выходе: null → 0. Поэтому в коде PDP ты можешь не бояться null в цене.

Это мощный приём: инвариант («цена всегда ≥0, 2 знака, не null») живёт в одном месте — в схеме поля, а не размазан по всем местам, где цену пишут или читают.

5. Access control — главная мысль урока

Все access-функции проекта собраны в src/access/index.ts — четыре однострочника:

export const ONLY_AUTHENTICATED = ({ req }) => !!req.user;
export const ONLY_ADMIN = ({ req }) => req.user?.collection === 'users';
export const ALLOW_ALL = () => true;
export const RESTRICTED_ALL = () => false;

В Products.ts:23 они расставлены по операциям: read: ALLOW_ALL (товары видят все), create/update: ONLY_AUTHENTICATED, delete: ONLY_ADMIN (удалять — только админ из коллекции users, не покупатель).

Но access умеет больше, чем true/false

Вот ключевой трюк Payload. Посмотри Orders.ts:27:

access: {
  read: ({ req }) => {
    if (req.user) return { customer: { equals: req.user.id } };  // ← не boolean, а ФИЛЬТР
    return false;
  },
},

Функция вернула не true, а объект-условие запроса. Payload понимает это так: «доступ есть, но только к строкам, где customer == текущий юзер». Этот фильтр он подмешивает в каждый запрос — и в Local API, и в REST, и в админке. Покупатель физически не может прочитать чужой заказ: не потому что мы добавили if в каждый эндпоинт, а потому что движок дописывает WHERE customer = ? на уровне доступа.

Почему это дорогого стоит. В классическом стеке (та же Java) «пользователь видит только свои заказы» — это ручной фильтр в сервисе, который легко забыть в новом эндпоинте → утечка данных. В Payload правило живёт в одной точке (access коллекции) и применяется ко всем путям доступа автоматически. Это row-level security декларативно.

Три формы возврата access-функции, которые надо знать:

ВернулаЗначит
trueполный доступ ко всем документам
falseдоступа нет вообще
{ ...where }доступ только к документам, подходящим под фильтр

6. Поменял схему — что делать дальше

Любая правка fields затрагивает БД и типы. Рабочий цикл в этом проекте:

# 1. правишь fields в коллекции # 2. dev-сервер на SQLite сам пушит изменение схемы (push) npm run dev # 3. перегенерировать TS-типы под новую форму данных npm run generate:types # обновит src/payload-types.ts # 4. (если правил admin-компоненты/блоки) — importmap npm run generate:importmap
Не забудь generate:types. Если добавил поле, но не перегенерировал типы — TS не будет знать о новом свойстве, и в коде PDP/карточки ты получишь ошибку «Property does not exist on type Product». src/payload-types.ts — автоген, руками не трогать (см. CLAUDE.md).
