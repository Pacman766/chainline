Payload Hooks: код на события жизненного цикла

Ты уже трогал хуки мимоходом: products/afterChange бил по /api/revalidate, поле price округлялось в field-hook. Теперь — системно. Хук = твой код, который Payload вызывает в определённый момент работы с документом: до записи, после записи, перед удалением, после логина.

Цель. Понять карту хуков и три вещи, которые отличают «работает» от «стреляет себе в ногу»: (1) что приходит в аргументах и чем отличаются типы хуков, (2) что нужно вернуть (и почему before* может мутировать данные, а after* — нет), (3) как не словить бесконечный цикл и не уронить операцию побочкой.

1. Обзор: где на таймлайне живут хуки

Мост от ExtJS. В ExtJS у store/model есть события: beforewrite, write, beforesync, и ты вешаешь на них листенеры. Payload hooks — то же самое, но на сервере и встроены в CRUD движка. Разница принципиальная: ExtJS-листенер на клиенте можно обойти (полез в API напрямую) — серверный хук Payload отработает при любом пути записи: админка, REST, GraphQL, Local API, seed. Это место для инвариантов, которые нельзя обойти.

Коллекционные хуки в порядке срабатывания на запись/чтение/удаление:

2. Разбор примера со сносками — orders/afterChange.ts

Лучший пример в проекте: при создании заказа отправить покупателю письмо. Каждая строка тут — отдельный урок. Цифры [n] — сноски под кодом.

import { CollectionAfterChangeHook } from 'payload';                // [1]

export const afterChange: CollectionAfterChangeHook = async ({
  doc,                                                          // [2]
  req: { payload },                                             // [3]
  operation,                                                    // [4]
}) => {
  if (operation !== 'create') return doc;                       // [5]
  const customerId = typeof doc.customer === 'object'            // [6]
    ? doc.customer.id : doc.customer;
  if (!customerId) return doc;                                  // [7]

  const customer = await payload.findByID({                       // [8]
    collection: 'customers', id: customerId,
  });
  try {                                                          // [9]
    await payload.sendEmail({
      to: customer.email,
      subject: `Заказ #${doc.id} принят`,
      html: `<p>Спасибо! Заказ на ${doc.total} ₽ принят.</p>`,
    });
  } catch (err) {
    payload.logger.error({ err }, 'Failed to send email');     // [10]
  }
  return doc;                                                   // [11]
};
  1. Типизированная сигнатура. CollectionAfterChangeHook — готовый тип из payload, он диктует, какие поля есть в аргументе. У каждого вида хука свой тип (CollectionBeforeChangeHook, CollectionBeforeDeleteHook…) с разным набором аргументов. Импортируешь нужный — и TS подсказывает, что доступно.
  2. doc — итоговый документ уже после записи в БД (с присвоенным id, с дефолтами). В before*-хуках вместо него работают с data (черновик до записи) — это разные вещи.
  3. req: { payload } — деструктурируешь из req экземпляр Payload, чтобы делать запросы (findByID, sendEmail, logger). В req также лежат req.user (кто инициировал) и req.context (см. сноску про seed ниже).
  4. operation — 'create' или 'update'. Один afterChange вызывается на обе операции, поэтому почти всегда первым делом разветвляешь по нему.
  5. Guard-выход. Письмо нужно только на создание заказа, не на каждое его редактирование. if (operation !== 'create') return doc — ранний выход. Заметь: возвращаем doc, а не undefined (сноска [11]).
  6. relationship приходит как id ИЛИ объект. doc.customer — это number (если depth:0) или populated {id, email,...} (если depth≥1). Тернарник нормализует к чистому id. Это та же дилемма «id vs объект», что ты уже разруливал на PDP с category.
  7. Defensive guard. Заказ без покупателя слать некому — выходим. Хук обязан переживать «грязные» данные.
  8. Догружаем связь. В doc может не быть email покупателя (если depth был 0 — там только id). Поэтому отдельный findByID за customers, чтобы достать customer.email.
  9. try/catch вокруг побочки — критично. Заказ уже сохранён (мы в afterChange). Если упадёт отправка почты, нельзя дать ошибке вырваться наружу — иначе клиент получит 500 на УЖЕ созданный заказ. Письмо — некритичная побочка, её провал не должен валить операцию.
  10. Логируем, а не глотаем. payload.logger.error — ошибка зафиксирована (можно потом разослать вручную), но операция продолжается. «Не глотать исключения» ≠ «дать им всё уронить».
  11. Возврат doc. Соглашение Payload: afterChange возвращает документ (тут без изменений). В before*-хуках возврат важнее — там возвращённый data и есть то, что запишется (см. практику).

Второй пример покороче — beforeDelete.ts (хук, который ОТМЕНЯЕТ операцию)

export const beforeDelete: CollectionBeforeDeleteHook = async ({ id, req: { payload } }) => {
  const orders = await payload.find({
    collection: 'orders',
    where: { 'items.product': { equals: id } },          // заказы, где есть этот товар
  });
  if (orders.totalDocs > 0)
    throw new Error('Нельзя удалить товар с активными заказами');  // ← отмена
};

Здесь главная идея before*-хуков-стражей: бросишь исключение — Payload отменит всю операцию и вернёт ошибку вызывающему (в админке всплывёт тостом). Это декларативная защита целостности данных: товар, на который ссылаются заказы, физически не дашь удалить — ни из админки, ни через API. Никакого «ON DELETE RESTRICT» в SQL руками — логика живёт в хуке.

Две ловушки, которые надо знать заранее:
• Бесконечный цикл. Если в afterChange сделать payload.update() того же документа — снова сработает afterChange → снова update → ∞. Лечится флагом в req.context: if (req.context.skip) return; … payload.update({..., context:{ skip:true }}). Ты уже видел этот приём как context.isSeeding — им seed глушит хук revalidate, чтобы массовая заливка не дёргала ревалидацию на каждый товар.
• Хук — это латентность операции. Всё, что ты await'ишь в before/afterChange, удлиняет ответ на запись. Тяжёлое (письма, внешние API) в идеале — в очередь/фон, а не блокирующим await. В учебном проекте ок, в проде — помни.

4. Практика 5 (пишешь сам)

Задача с реальным смыслом для магазина: сейчас total заказа приходит с клиента (см. api/checkout) — это дыра: клиент может подделать сумму. Закроем её серверным хуком, который пересчитывает total из items и не доверяет присланному значению.

  1. Создай src/hooks/orders/beforeChange.ts. Тип — CollectionBeforeChangeHook. В аргументах тебе нужен data (а НЕ doc — почему? объясни словами).
  2. Внутри посчитай сумму по data.items: Σ (item.price × item.quantity). Перезапиши data.total = посчитанным значением и верни data.
  3. Подключи хук в Orders.ts → hooks: { beforeChange: [beforeChange], afterChange: [...] } (afterChange не трогай).
  4. Проверь: оформи заказ через витрину/checkout, передай заведомо неверный total (можно временно захардкодить в запросе) — в БД и в письме должна оказаться пересчитанная серверная сумма, а не присланная.
  5. На подумать (без кода): почему этот хук правильнее как beforeChange на коллекции, а не как field-hook на поле total? Подсказка: откуда field-hook на total взял бы items?
