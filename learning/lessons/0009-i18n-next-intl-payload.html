Урок 9 · Next.js + Payload — i18n · обзор → разбор → практика

Локализация: два разных механизма под одной кнопкой

«Сменить язык» звучит как одна задача, а на деле это две независимые проблемы с разными инструментами. Самая частая ошибка — пытаться решить их одним. Этот урок ставит ментальную модель, а потом разбирает целевую настройку на твоём проекте: next-intl для интерфейса + встроенная локализация Payload для контента.

Цель. Понять три вещи: (1) UI-обвязка ≠ контент — почему это разные механизмы и где живёт каждая строка; (2) как локаль попадает в URL-префикс /ru /en и почему из-за этого весь (frontend) переезжает под сегмент [locale]; (3) почему локаль из URL — единственный источник правды, который обязан управлять и интерфейсом, и параметром locale в запросах к Payload.

1. Обзор: две поверхности, два инструмента

Мост от ExtJS. В ExtJS были locale-паки (ext-lang-ru.js) — они переводили служебные строки фреймворка (кнопки грида, «Loading…»). Но данные в сторе ты переводил сам, на бэке. Та же граница и здесь: next-intl = locale-пак для твоего UI, Payload localization = перевод данных. Никто из них не делает работу другого.

У любой строки на странице есть владелец. Спроси: «эту фразу пишет разработчик в коде или контент-менеджер в админке?» — ответ выбирает механизм.

Ключевой вывод урока. Эти два механизма соединяет одна переменная — текущая локаль. В нашем дизайне её источник — префикс URL. Если UI показывает /en, а к Payload ты случайно сходил без locale:'en' — получишь английскую обвязку с русскими названиями товаров. Рассинхрон локали — главный баг этой фичи.

2. Разбор целевой настройки со сносками

Кода ещё нет — строим его в этой фиче (#18). Ниже целевые файлы с построчными сносками, привязанные к твоей структуре (src/app/(frontend), parallel route products/@sidebar, payload.config.ts).

2a. Контент: локализация в Payload

// src/payload.config.ts — добавляем блок localization
export default buildConfig({
  // ...
  localization: {                       [1]
    locales: ['ru', 'en'],          [2]
    defaultLocale: 'ru',             [3]
    fallback: true,                  [4]
  },
})
// src/collections/Products.ts — помечаем поля
fields: [
  { name: 'name', type: 'text', localized: true },        [5]
  { name: 'description', type: 'richText', localized: true },
  { name: 'price', type: 'number' },                       [6]
  { name: 'sku', type: 'text' },
]
  1. Один блок в конфиге включает мультиязычность для всего проекта. До него поля одноязычные.
  2. Список локалей. Порядок не важен; важно, что это контентные локали Payload — отдельные от локалей next-intl (их надо держать одинаковыми, см. [12]).
  3. defaultLocale: 'ru' — то, к чему ведёт fallback. Это закреплено в ADR 0002 и CONTEXT.md: русский — базовый.
  4. fallback: true — если у поля нет значения на запрошенной локали, отдаётся значение defaultLocale. Так англоязычный покупатель не увидит пустое описание (наш ответ на грилле).
  5. localized: true — теперь Payload хранит отдельное значение на каждую локаль и в админке рисует вкладки RU/EN над полем. В БД это одна колонка JSON по локалям (деталь хранения, тебе её знать не обязательно).
  6. Цена и sku не локализуются — число одно на все языки. Локализуй только то, что реально различается по языку. Это сознательный выбор, а не «на всякий случай».
  7. Миграция: существующие одноязычные строки Payload при включении localization присвоит defaultLocale (RU). Сид дальше должен заполнять обе локали.
Как теперь выглядит запрос к контенту 
const payload = await getPayload({ config })
const { docs } = await payload.find({
  collection: 'products',
  locale: locale,            // 'ru' | 'en' — берётся из URL, см. [12]
  fallbackLocale: 'ru',      // подстраховка к config.fallback
})

2b. UI: next-intl и префиксный роутинг

❗ DISCLAIMER: next-intl активно развивается (актуальная major-линия ~v4). Перед установкой сверимся с доками через Context7 — API defineRouting/createNavigation ниже стабилен в v3.22+/v4, но версии проверим на старте практики.
// src/i18n/routing.ts
import { defineRouting } from 'next-intl/routing'
export const routing = defineRouting({       [8]
  locales: ['ru', 'en'],
  defaultLocale: 'ru',
})
// middleware.ts (корень)
import createMiddleware from 'next-intl/middleware'
import { routing } from '@/i18n/routing'
export default createMiddleware(routing)        [9]
export const config = { matcher: ['/((?!api|admin|_next|.*\\..*).*)'] }  [10]
// src/app/[locale]/(frontend)/layout.tsx — упрощённо
import { NextIntlClientProvider } from 'next-intl'
import { setRequestLocale, getMessages } from 'next-intl/server'

export default async function LocaleLayout({ children, params }) {
  const { locale } = await params           [11]
  setRequestLocale(locale)                    [11]
  const messages = await getMessages()
  return (
    <NextIntlClientProvider messages={messages}>
      {children}
    </NextIntlClientProvider>
  )
}
// messages/ru.json            // messages/en.json
{ "hero": {                     { "hero": {
  "cta": "Смотреть каталог"        "cta": "Browse catalog"
} }                            } }
// в компоненте
import { useTranslations } from 'next-intl'
const t = useTranslations('hero')
<Link href="/products">{t('cta')}</Link>     [13]
  1. routing.ts — единое описание локалей для UI. Держи список идентичным локалям Payload из [2], иначе URL /en будет, а контент на en — нет.
  2. Middleware определяет локаль из URL (или редиректит /products → /ru/products) и проставляет её в запрос. Это «вход» всей i18n-цепочки.
  3. Главная ловушка для нашего проекта: matcher исключает api и admin. Иначе middleware начнёт префиксовать админку Payload и REST-роуты — всё сломается. Запомни этот negative-lookahead.
  4. В Next 15 params — это Promise (мы это уже видели в /orders/[id]). setRequestLocale нужен, чтобы серверные компоненты ниже могли рендериться статически и знали локаль.
  5. Сцепка двух механизмов. Та же locale из params, что кормит next-intl, обязана уехать в payload.find({ locale }) (см. 2a). Это и есть «единый источник правды» из цели урока.
  6. useTranslations читает активную локаль из провайдера — в самом компоненте никаких if (locale === 'ru'). Ключи (hero.cta), а не строки, — переключение языка не трогает JSX.

2c. Что переезжает физически

Префиксный роутинг означает реструктуризацию дерева. Было → станет:

src/app/(frontend)/page.tsx        →  src/app/[locale]/(frontend)/page.tsx
src/app/(frontend)/products/...    →  src/app/[locale]/(frontend)/products/...
src/app/(frontend)/products/@sidebar  →  ...едет вместе, parallel route сохраняется
Аккуратно с (payload) и api. Под [locale] едет только сторфронт (frontend). Группа админки (payload) и app/api остаются на месте — их и исключает matcher [10].

4. Практика 7 (первый срез — пишешь сам)

Не строим всё сразу. Первый вертикальный срез — только контентный механизм, чтобы увидеть его в админке без возни с роутингом:

  1. В src/payload.config.ts добавь блок localization (locales ['ru','en'], default 'ru', fallback: true).
  2. Пометь localized: true на Products.name и Products.description. Не трогай price/sku — подумай почему (сноска [6]).
  3. npm run generate:types → загляни в payload-types.ts: как изменился тип name? (подсказка: посмотри, появилась ли разница для локализованных полей).
  4. Запусти npm run dev, открой товар в /admin — над name/description должны появиться вкладки локалей. Заполни EN-название одного товара.
  5. На подумать (ответишь словами): существующие 6 товаров были одноязычными. В какую локаль Payload положил их старые названия при включении localization и почему именно туда? (сноска [7])

UI-механизм (next-intl, роутинг, свитчер) — следующий срез, после того как увидишь контентную локализацию живьём. Так фича #18 разбивается на понятные шаги, а не на один большой коммит.

Споткнёшься или будет неочевидно (особенно п.3 — тип локализованного поля, и п.5) — спрашивай. После практики дам UI-срез: routing.ts + middleware + перенос под [locale] + свитчер языка.
